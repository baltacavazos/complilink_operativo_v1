# Consenso multi-IA y hallazgos técnicos para /acceso

## Consenso de OpenAI, Grok y Gemini

La recomendación convergente es mantener **una sola ruta primaria** de acceso: **correo + código OTP**. Los tres coinciden en eliminar ruido visual y rutas secundarias prominentes.

| Tema | Consenso útil |
| --- | --- |
| Ruta principal | Correo y código OTP como flujo único visible |
| Eliminar | Explicación del CEO, aviso de estabilidad del dominio, estados detallados del OTP, errores largos, opciones secundarias prominentes |
| Mantener | Campo de correo, CTA para pedir código, campo de código, CTA para verificar |
| Usuario recurrente | Recordar el último correo y prellenarlo para reducir fricción |
| Manejo de error | Mensaje corto y accionable; si expira, sugerir o disparar reenvío claro |
| Estructura | Paso 1: correo. Paso 2: código. Nada más como protagonista |

## Hallazgos del backend

- `completeEmailLogin()` ya crea una **cookie de sesión de 1 año** con `createAppSessionForUser()`.
- La cookie usa opciones compartidas del dominio y se limpia el challenge OTP al completar el acceso.
- Por tanto, el problema actual percibido es principalmente de **UX y claridad del flujo**, no de falta de sesión larga en el backend.
- El backend sí tiene restricciones reales que la UI debe tratar con sobriedad: TTL de 10 minutos para el código, cooldown de 60 segundos y límite de 5 solicitudes por hora.

## Decisiones de implementación

1. Dejar un hero mínimo y eliminar bloques informativos redundantes.
2. Ocultar Google/Manus como rutas principales en dominio público; si se conservan, que sea de forma secundaria y discreta o fuera de la vista principal.
3. Convertir el formulario en un flujo de dos etapas muy limpio.
4. Recordar el último correo usado en el dispositivo para prellenarlo.
5. Simplificar mensajes de error y de estado.
6. Valorar autoenvío al completar 6 dígitos si no introduce fragilidad.
7. Corregir primero el error de compilación activo en `Auditar.tsx` para poder validar en móvil.
